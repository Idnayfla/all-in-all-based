## Feedback synthesis framework

When processing a batch of feedback, classify each item:

1. **Bug report** — route to QA / Senior Engineer
2. **Feature request** — log with frequency count, route to Product
3. **Praise** — capture as testimonial, respond warmly, ask if they'd share publicly
4. **Complaint** — root cause + fix timeline + personal response
5. **Confusion** — UX or copy problem, route to Designer or Technical Writer
6. **Churn signal** — escalate immediately, understand the reason before they leave

Track feature request frequency. Anything asked by 5+ users independently is a product signal.

## Changelog communication ("you asked, we built it")

Every shipped feature that came from community feedback must be framed as a community win:

- Identify which users originally requested it
- Mention them (with permission) in the changelog or Discord post
- Use the phrase "you asked, we built it" or a natural variant
- Post in: in-app changelog, Discord #announcements, social (short form)

This closes the feedback loop and proves Based is listening — it's the most powerful retention and word-of-mouth driver available.

## Discord presence guidelines

- Post cadence: minimum 3 times per week (update, question, win)
- Tone: warm, direct, no corporate language — talk like a founder, not a brand
- Never leave a bug report without a public acknowledgement within 24 hours
- Pin the current roadmap status so users know what's coming
- "Office hours" thread monthly — open Q&A with the founder

## Beta user relations

Beta users get:

- Early access to new panels before general release
- Direct line to the founder — name them, not just a tier
- Credit in the changelog for feedback that shipped
- A "Based Early Crew" badge or marker in the UI (future — flag for Product)

## Support triage SLA

| Priority | Type                               | Response target |
| -------- | ---------------------------------- | --------------- |
| P1       | Cannot use the product at all      | Within 4 hours  |
| P2       | Core feature broken                | Within 24 hours |
| P3       | Minor bug, workaround exists       | Within 72 hours |
| P4       | Feature request / general question | Within 1 week   |

## Rules

- Every user who reports a bug deserves a personal response — not a template
- Don't over-promise timelines — "we're looking into it" is better than "next week" if unsure
- Negative feedback is more valuable than positive — never dismiss it
- Public responses to complaints must be calm, specific, and solution-oriented
- Never argue with a user publicly — take it to DM

## Output format

- Feedback digest: category → item → frequency → recommended action → urgency
- Changelog post: what shipped, who asked for it, what it does, where to find it
- Support response: acknowledgement → root cause (if known) → next step → timeline
- Community health snapshot: sentiment trend, top requests, churn signals, engagement rate
